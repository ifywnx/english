Quy tắc dùng tiếng Anh trong team IT

IT English không chỉ là nhớ từ vựng. Quan trọng hơn là hiểu cụm từ nào dùng cho code, cụm nào dùng cho họp, cụm nào dùng khi mô tả sự cố hoặc thảo luận kỹ thuật.

1. Code với communication là hai thứ khác nhau

Khi nói về code, bạn có thể ngắn và technical. Khi nói với PM, QA hoặc khách hàng, cần rõ ràng, tránh jargon quá sâu và luôn giải thích tác động.

2. Dùng động từ chính xác

Trong IT English, các động từ như deploy, refactor, rollback, monitor, debug, prioritize rất quan trọng vì chúng mô tả hành động cụ thể.

3. Mỗi tình huống một tone

Standup thì ngắn gọn, code review thì góp ý rõ ràng, incident thì ưu tiên mức độ nghiêm trọng, phỏng vấn thì nói có cấu trúc và có kết quả.

4. Ưu tiên rõ hơn là dài

“The API is broken because…” tốt hơn “Something seems wrong”. Hãy mô tả vấn đề, nguyên nhân, cách xử lý và thời gian hoàn tất.

Lưu ý: Trong môi trường IT, sự rõ ràng giúp tiết kiệm thời gian. Một câu tốt là câu vừa đúng chuyên môn vừa dễ hiểu với người cùng team.

Những cặp từ dễ nhầm trong IT

Phần này giúp bạn tránh dịch word-by-word và dùng đúng thuật ngữ theo ngữ cảnh.

Khái niệmKhác nhauVí dụ dùng thật
Frontend / BackendFrontend là phần người dùng nhìn thấy; backend là logic, server, database phía sau.Frontend fetches data from the backend API.

Hội thoại mẫu

Hội thoại trong team IT

Giữ phần dialog cũ nhưng mở rộng thành các tình huống rất thật để bạn đọc là hiểu cách nói.

Daily standup

Scrum Master
Good morning, team. Let's keep it brief. What did you work on yesterday?
Chào cả team, mình giữ ngắn thôi nhé. Hôm qua các bạn làm gì?
Developer
I finished the authentication module and pushed it to staging.
Tôi đã xong module xác thực và đẩy lên staging.

Các cụm rất IT: keep it brief, pushed it to staging, blockers.

Code review

Reviewer
I left some comments on your pull request. Can we go through them?
Mình đã để lại vài comment trên pull request của bạn. Mình xem qua nhé?
Author
Sure. I’ll split the data transformation into a separate utility.
Được thôi. Tôi sẽ tách phần xử lý dữ liệu thành utility riêng.

Đây là cách nói tự nhiên trong review: góp ý thẳng nhưng lịch sự.

Phỏng vấn tech

Hỏi phỏng vấn và cách trả lời

Phần này giữ danh sách câu hỏi cũ, nhưng thêm cách trả lời theo cấu trúc để bạn học được thật.

How do you handle tight deadlines?

Trả lời theo công thức: chia task, ưu tiên impact, communicate sớm, giữ chất lượng. Có thể nói về việc dùng sprint planning hoặc timeboxing.

How do you ensure code quality?

Nói về code review, unit test, linting, CI/CD, clean code, documentation, và cách bạn cân bằng tốc độ với chất lượng.

Explain SQL vs NoSQL

Nêu quan hệ dữ liệu, ACID, schema linh hoạt, horizontal scaling, eventual consistency và use case thực tế.

How would you design a scalable system?

Mô tả load balancing, caching, microservices, database scaling, CDN, queue, monitoring, và rollback plan.

Lỗi hay gặp

Người Việt hay sai ở đâu?

Đây là phần giúp module thực sự học được, không chỉ là trang glossary.

1. Dịch thẳng từng từ

“Make code”, “do deploy”, “close bug” nghe không tự nhiên. Hãy học collocation đúng: write code, deploy code, fix a bug.

2. Không phân biệt tình huống

Standup khác code review, khác incident report, khác phỏng vấn. Một câu dùng đúng ngữ cảnh mới là câu tốt.

3. Nói quá chung chung

“There is a problem” chưa đủ. Hãy mô tả cụ thể: where, what happened, impact, current status, next action.

4. Thiếu động từ hành động

Hãy ưu tiên các verb như implement, refactor, monitor, investigate, prioritize, troubleshoot thay vì dùng từ mờ.

Học tiếp

Kết nối module liên quan

IT English mạnh hơn khi đi cùng grammar, collocations và reading practice.

Grammar cần học kèm

Modal verbs, passive voice, conditionals, relative clauses và reported speech giúp bạn diễn đạt kỹ thuật mạch lạc hơn.

Vocabulary cần học kèm

Collocations, confusing words, linking words và academic words giúp câu nói nghe chuyên nghiệp hơn.
